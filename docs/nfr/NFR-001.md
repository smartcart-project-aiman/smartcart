## 1. Purpose and Scope

This document defines every non-functional requirement (NFR) for
SmartCart. An NFR is a quality attribute: a constraint on *how* the
system must behave, as distinct from *what* it must do.

Every NFR has:

- A unique stable ID in the format `NFR-<CATEGORY>-<NUMBER>`
- A measurable threshold (never qualitative)
- A verification method (how it will be confirmed)
- Tier applicability (T1 / T2 / T3 / All)
- A trace back to a PRD section or system prompt convention where applicable

If an implementation choice cannot satisfy an NFR in this document,
the NFR takes precedence and the implementation must change or an ADR
must be raised requesting a formal exception with written rationale.

## 2. Service Tier Map

Tier determines the rigor applied to availability, alerting, and
disaster recovery. Every service belongs to exactly one tier. Tier
assignment is not a subjective preference — it follows from the blast
radius of a service outage on customer-facing revenue.

| Tier | SLO       | Monthly Error Budget    | Services                                            |
|------|-----------|-------------------------|-----------------------------------------------------|
| T1   | 99.9 %    | 43 min 49 sec           | Auth, Order, Payment, Inventory                     |
| T2   | 99.5 %    | 3 hr 39 min 18 sec      | Product Catalog, Cart, Search, Notification, API Gateway |
| T3   | 99.0 %    | 7 hr 18 min 17 sec      | Recommendation, Chatbot, Review, Analytics, Fraud Detection |

### 2.1 Error Budget Derivation

For a 30-day month (43 200 minutes):

```
T1: 43 200 × (1 - 0.999) =  43.2  min downtime allowed
T2: 43 200 × (1 - 0.995) = 216.0  min downtime allowed
T3: 43 200 × (1 - 0.990) = 432.0  min downtime allowed
```

Error budget is measured as the aggregate of any period where the
service's error rate exceeds its SLO threshold, measured at the
Prometheus scrape level (15 s intervals) over a rolling 30-day window.

## 3. Performance NFRs

Performance NFRs specify the maximum acceptable latency and throughput
at stated percentiles and load levels. "At load" means the platform is
sustaining 100 RPS with burst headroom to 300 RPS (NFR-PERF-030).

All latency measurements are **end-to-end wall-clock time** from the
client request entering the API Gateway to the last byte of the response
leaving the gateway, unless the NFR explicitly states otherwise.

| Measurement method | k6 load test at 100 RPS; Grafana histogram     |
| Alert threshold    | p99 > 400 ms for 5 consecutive minutes → PagerDuty |

## 4. Reliability NFRs

Reliability NFRs govern failure modes, recovery behaviour, and the
correctness guarantees the system makes under partial failure.

### NFR-REL-001 — Idempotency on Write Endpoints
| Field              | Value                                          |
|--------------------|------------------------------------------------|
| Threshold          | All state-mutating endpoints (POST, PUT, PATCH) accept an `Idempotency-Key` header; repeat requests with the same key return the cached response without re-executing the operation |
| Applies to         | All services (All tiers)                       |
| PRD trace          | FR-CHECKOUT-003 (payment idempotency key), FR-CHECKOUT-001 |
| Measurement method | Integration test: POST with key K → response R; POST again with K → same R; verify no duplicate DB record |
| Key storage        | Redis with TTL = 24 h; key format: `idempotency:<service>:<key>` |

### NFR-REL-002 — Kafka Consumer Idempotency
| Field              | Value                                          |
|--------------------|------------------------------------------------|
| Threshold          | All Kafka consumers tolerate at-least-once delivery; processing the same message twice must produce the same system state as processing it once |
| Applies to         | All event consumers (All tiers)                |
| PRD trace          | System prompt Section 12                       |
| Measurement method | Testcontainers integration test: publish duplicate event → assert idempotent outcome; check via DB query that no duplicate record exists |
| Implementation     | Deduplication via `messageId` stored in a processed-events table per consumer group; or natural idempotency via upsert semantics |

### NFR-REL-003 — Saga Compensation Coverage
| Field              | Value                                          |
|--------------------|------------------------------------------------|
| Threshold          | Every forward step in every saga has a documented and tested compensation step |
| Applies to         | Order Service saga (T1)                        |
| PRD trace          | FR-CHECKOUT-004 saga outcomes table; System prompt Section 12 |
| Measurement method | Each compensation path covered by a dedicated integration test that forces the failure condition and asserts full rollback |
| Anti-pattern named | "Saga without compensation" — a saga step that can fail with no recovery path is a guaranteed data inconsistency event |

### NFR-REL-004 — Inventory Reservation TTL
| Field              | Value                                          |
|--------------------|------------------------------------------------|
| Threshold          | Inventory reservations auto-expire and release after 15 minutes if payment is not completed |
| Applies to         | Inventory Service (T1)                         |
| PRD trace          | FR-CHECKOUT-002 acceptance criteria            |
| Measurement method | Integration test: create reservation → advance clock 15 min (or set short TTL in test config) → assert stock returned to available quantity |
| Implementation     | TTL enforced by a scheduled job or Redis TTL on reservation key — not by application logic dependent on payment callback |

### NFR-REL-005 — Transactional Outbox for Critical Events
| Field              | Value                                          |
|--------------------|------------------------------------------------|
| Threshold          | Events that must not be lost (order state changes, payment results, inventory commits) are written to an outbox table in the same DB transaction as the business data change, then forwarded to Kafka by a dedicated relay process |
| Applies to         | Order (T1), Payment (T1), Inventory (T1)       |
| PRD trace          | Glossary "Outbox" definition; ADR-005          |
| Measurement method | Chaos test: kill Kafka broker immediately after a business write — assert event is eventually delivered once broker recovers; assert no duplicate or lost messages |
| Anti-pattern named | "Dual write" — writing to DB then publishing to Kafka as two separate operations; if the broker call fails the event is silently lost |

### NFR-REL-006 — Retry Policy
| Field              | Value                                          |
|--------------------|------------------------------------------------|
| Threshold          | All synchronous outbound calls (HTTP, gRPC) retry on transient failures using exponential backoff with jitter; maximum 3 attempts before circuit opens |
| Applies to         | All services (All tiers)                       |
| PRD trace          | System prompt Section 12                       |
| Measurement method | Unit test: mock downstream returning 503 three times → assert caller retried exactly 3 times with increasing delay; assert circuit breaker opened after threshold |
| Backoff formula    | `delay = base × 2^attempt + rand(0, base)` where `base = 100 ms`; max cap 5 s |
| Circuit breaker    | Half-open after 30 s; close after 3 consecutive successes |

### NFR-REL-007 — Dead-Letter Queue (DLQ) Monitoring
| Field              | Value                                          |
|--------------------|------------------------------------------------|
| Threshold          | Every Kafka topic has a corresponding DLQ topic; Prometheus alert fires when DLQ depth > 0 for > 5 minutes |
| Applies to         | All Kafka consumers (All tiers)                |
| PRD trace          | FR-ADMIN-001 (DLQ depth in dashboard); Glossary "DLQ" |
| Measurement method | Prometheus alert rule on `kafka_consumergroup_lag` for DLQ topics; integration test forces an unprocessable message and asserts it lands in DLQ |
| Naming convention  | DLQ topic: `<original-topic>.dlq`              |

### NFR-REL-008 — Transactional Money Operations
| Field              | Value                                          |
|--------------------|------------------------------------------------|
| Threshold          | Every operation that reads and writes a monetary balance or amount is executed within a database transaction with appropriate isolation level; no money operation may partially succeed |
| Applies to         | Payment (T1), Order (T1), Inventory (T1)       |
| PRD trace          | System prompt Section 12                       |
| Measurement method | Integration test: simulate crash mid-transaction → assert atomicity via DB query |
| Isolation level    | READ COMMITTED minimum; SERIALIZABLE for inventory deduction |

### NFR-REL-009 — Webhook HMAC Signature Verification
| Field              | Value                                          |
|--------------------|------------------------------------------------|
| Threshold          | Payment Service must verify the HMAC-SHA256 signature on every incoming webhook from the mock payment gateway before processing the payload; unverified webhooks must return 401 and be logged as security events |
| Applies to         | Payment Service (T1)                           |
| PRD trace          | FR-CHECKOUT-003 acceptance criteria            |
| Measurement method | Unit test: valid signature → processed; tampered signature → 401 + security log entry |

### NFR-REL-010 — Money Type Enforcement
| Field              | Value                                          |
|--------------------|------------------------------------------------|
| Threshold          | `float` and `double` are prohibited for all monetary values throughout the entire codebase, in all languages |
| Applies to         | All services (All tiers)                       |
| PRD trace          | System prompt Section 10 "Data & Money" conventions |
| Enforcement        | Java → `java.math.BigDecimal`; TypeScript → `Decimal` from `decimal.js`; Python → `decimal.Decimal`; PostgreSQL column type → `NUMERIC(19,4)`; MongoDB → `Decimal128`; JSON wire format → amount as string e.g. `"1250.0000"` |
| Measurement method | SAST rule (SonarQube / Semgrep) blocks any use of `float` or `double` in a file that handles monetary amounts; enforced in CI |
| Anti-pattern named | "Float money" — storing `₹1250.00` as `1250.0` in a double-precision float introduces rounding drift that compounds across millions of transactions |

### NFR-REL-011 — Graceful Shutdown
| Field              | Value                                          |
|--------------------|------------------------------------------------|
| Threshold          | Services drain in-flight requests within 30 s of receiving SIGTERM before exiting |
| Applies to         | All services (All tiers)                       |
| PRD trace          | System prompt Section 10 "Operability"         |
| Measurement method | Integration test: send SIGTERM during active request processing → assert request completes before process exits; assert no 5xx returned to client |

### NFR-REL-012 — Vendor Data Isolation
| Field              | Value                                          |
|--------------------|------------------------------------------------|
| Threshold          | A vendor's API request that attempts to read or write another vendor's resource must receive 403 Forbidden; the attempt must be logged as a security event |
| Applies to         | Product Catalog (T2), Order (T1), Analytics (T3) |
| PRD trace          | FR-AUTH-006 (RBAC), FR-ORDER-002 (vendor sees only own sub-orders) |
| Measurement method | Integration test: vendor A's JWT used to access vendor B's resource → assert 403 + security log |

## 5. Availability NFRs

Availability NFRs specify the SLO per tier, the required degradation
paths when dependencies fail, and the circuit-breaking behaviour that
prevents cascade failures.

### NFR-AVAIL-001 — T1 Availability SLO
| Field              | Value                                          |
|--------------------|------------------------------------------------|
| Threshold          | Auth, Order, Payment, Inventory achieve 99.9 % availability over any rolling 30-day window |
| Measurement method | Prometheus `up` metric + synthetic probe (Blackbox Exporter) hitting `/health/ready` every 30 s; SLO dashboard in Grafana |
| Error budget       | 43 min 49 sec per 30-day month                 |
| Alert              | SLO burn-rate alert: multi-window (1 h fast burn + 6 h slow burn) |

### NFR-AVAIL-002 — T2 Availability SLO
| Field              | Value                                          |
|--------------------|------------------------------------------------|
| Threshold          | Product Catalog, Cart, Search, Notification, API Gateway achieve 99.5 % availability over any rolling 30-day window |
| Measurement method | Same as NFR-AVAIL-001                          |
| Error budget       | 3 hr 39 min 18 sec per 30-day month            |
| Alert              | SLO burn-rate alert: multi-window              |

### NFR-AVAIL-003 — T3 Availability SLO
| Field              | Value                                          |
|--------------------|------------------------------------------------|
| Threshold          | Recommendation, Chatbot, Review, Analytics, Fraud Detection achieve 99.0 % availability over any rolling 30-day window |
| Measurement method | Same as NFR-AVAIL-001                          |
| Error budget       | 7 hr 18 min 17 sec per 30-day month            |

### NFR-AVAIL-013 — Notification Service Failure Isolation
| Field              | Value                                          |
|--------------------|------------------------------------------------|
| Threshold          | If the Notification Service is down or its Kafka consumer is failing, the Order saga completes successfully; no order must fail due to a notification delivery failure |
| Applies to         | Order Service (T1), Notification Service (T2)  |
| PRD trace          | FR-NOTIF-001 (WebSocket "non-blocking"); FR-NOTIF-002 ("Email delivery failure does not fail the triggering business operation") |
| Measurement method | Chaos test: stop Notification container → place order → assert order reaches CONFIRMED status and inventory is committed |
| Implementation     | Notification events are fire-and-forget Kafka publishes; the saga does not await ACK from Notification service |
| Anti-pattern named | "Tight coupling to non-critical dependency" — if Order awaited a Notification callback before advancing saga state, every email provider outage would halt all orders |

### NFR-AVAIL-014 — Recommendation Service Failure Isolation
| Field              | Value                                          |
|--------------------|------------------------------------------------|
| Threshold          | If the Recommendation Service is unavailable, the homepage renders with a "popular items" fallback; the customer-facing page must not return an error |
| Applies to         | Recommendation Service (T3)                   |
| PRD trace          | FR-DISC-004 (cold start fallback)              |
| Measurement method | Integration test: Recommendation service returns 503 → homepage still returns 200 with fallback content |

### NFR-AVAIL-015 — Inventory Service Failure — Checkout Behaviour
| Field              | Value                                          |
|--------------------|------------------------------------------------|
| Threshold          | If Inventory Service is unavailable at checkout, checkout fails fast (within 2 s) with a descriptive error; the cart is preserved intact for retry |
| Applies to         | Inventory Service (T1), Cart Service (T2)      |
| PRD trace          | System prompt Section 12 NFR-AVAIL-015         |
| Measurement method | Integration test: Inventory unreachable → checkout returns 503 within 2 s → GET cart still returns all items |

### NFR-AVAIL-016 — Payment Service Failure — Checkout Behaviour
| Field              | Value                                          |
|--------------------|------------------------------------------------|
| Threshold          | If Payment Service is unavailable at checkout, checkout fails fast with a 503 and returns the idempotency key so the client can retry safely without risk of double-charge |
| Applies to         | Payment Service (T1), Order Service (T1)       |
| PRD trace          | System prompt Section 12 NFR-AVAIL-016         |
| Measurement method | Integration test: Payment service down → checkout returns 503 + `Idempotency-Key` header within 2 s |

### NFR-AVAIL-017 — Fraud Detection Fail-Open
| Field              | Value                                          |
|--------------------|------------------------------------------------|
| Threshold          | If the Fraud Detection Service is unavailable or does not publish a score within 10 s, the order proceeds as LOW-risk; fraud scoring failure must never block order completion |
| Applies to         | Fraud Detection Service (T3)                   |
| PRD trace          | FR-FRAUD-001 ("Fraud scoring failure is non-blocking") |
| Measurement method | Integration test: Fraud Detection container stopped → order reaches CONFIRMED status |
| Design note        | Fail-open is the correct default here because false negatives (missed fraud) are recoverable via Admin review; false positives (good orders blocked) are revenue-destroying and irreversible |

### NFR-AVAIL-018 — Circuit Breaker — Downstream Dependencies
| Field              | Value                                          |
|--------------------|------------------------------------------------|
| Threshold          | Every synchronous outbound call is wrapped by a circuit breaker; circuit opens after 5 consecutive failures or 50 % error rate over a 10 s window; half-open after 30 s |
| Applies to         | All services (All tiers)                       |
| Measurement method | Prometheus metric `circuit_breaker_state` (CLOSED / OPEN / HALF_OPEN) per dependency; alert on OPEN state |

## 6. Security NFRs

Security NFRs are hard gates. An NFR in this section failing in CI
is a blocking condition — the build does not pass and the PR cannot
be merged.

### NFR-SEC-001 — JWT Required on All Authenticated Endpoints
| Field              | Value                                          |
|--------------------|------------------------------------------------|
| Threshold          | Every endpoint that is not explicitly listed as public must require a valid RS256-signed JWT in the `Authorization: Bearer <token>` header; requests without a valid JWT return 401 |
| Applies to         | All services (All tiers)                       |
| PRD trace          | FR-AUTH-006, Section 2.1 "Auth-first, no guests" |
| Measurement method | Integration test per service: unauthenticated request to protected endpoint → 401 |

### NFR-SEC-002 — Token TTL Enforcement
| Field              | Value                                          |
|--------------------|------------------------------------------------|
| Threshold          | Access token TTL ≤ 15 minutes; Refresh token TTL ≤ 7 days; refresh token is rotated (invalidated and replaced) on every use |
| Applies to         | Auth Service (T1)                              |
| PRD trace          | FR-AUTH-003, FR-AUTH-004                       |
| Measurement method | Unit test: decode JWT → assert `exp - iat ≤ 900 s`; integration test: use refresh token twice → second use returns 401 |

### NFR-SEC-003 — RBAC at Service Layer
| Field              | Value                                          |
|--------------------|------------------------------------------------|
| Threshold          | Role-Based Access Control is enforced at the service's own request handler — not solely at the API Gateway or UI. A CUSTOMER JWT calling a VENDOR-only endpoint returns 403. A VENDOR JWT calling an ADMIN-only endpoint returns 403. |
| Applies to         | All services (All tiers)                       |
| PRD trace          | FR-AUTH-006                                    |
| Measurement method | Integration test per service: CUSTOMER JWT → VENDOR endpoint → 403; VENDOR JWT → ADMIN endpoint → 403 |

### NFR-SEC-004 — Vendor Tenant Isolation
| Field              | Value                                          |
|--------------------|------------------------------------------------|
| Threshold          | A vendor's JWT must not be usable to read or write another vendor's data; cross-vendor access returns 403 and generates a security log entry with `userId` and attempted resource |
| Applies to         | Product Catalog (T2), Order (T1), Analytics (T3) |
| PRD trace          | FR-AUTH-006, FR-ORDER-002                      |
| Measurement method | Integration test: vendor A's JWT → GET vendor B's product → 403 + log entry |

### NFR-SEC-007 — Password Hashing
| Field              | Value                                          |
|--------------------|------------------------------------------------|
| Threshold          | All passwords are hashed using bcrypt with cost ≥ 12, or argon2id with parameters meeting OWASP recommendations; plaintext passwords must never be stored or logged |
| Applies to         | Auth Service (T1)                              |
| PRD trace          | FR-AUTH-001 acceptance criteria                |
| Measurement method | Unit test: assert stored hash is not plaintext; assert bcrypt cost factor ≥ 12; SAST rule blocks any log statement containing "password" |

### NFR-SEC-008 — Rate Limiting
| Field              | Value                                          |
|--------------------|------------------------------------------------|
| Threshold          | Unauthenticated endpoints: 100 requests/minute per IP; Authenticated endpoints: 1 000 requests/minute per userId; Login endpoint: 10 failed attempts per 15 minutes per IP, then 429 |
| Applies to         | API Gateway (T2) — enforced centrally          |
| PRD trace          | FR-AUTH-003 rate limiting criteria; System prompt Section 12 NFR-SEC-026 |
| Measurement method | k6 test: burst 150 req/min unauthenticated → assert 429 responses after threshold; Prometheus metric `rate_limit_hit_total` |

### NFR-SEC-009 — JWKS Local Validation
| Field              | Value                                          |
|--------------------|------------------------------------------------|
| Threshold          | Each service validates JWT signatures locally using the Auth Service's JWKS endpoint (cached with a 5-minute TTL); no service makes a per-request call to Auth Service for token validation |
| Applies to         | All services (All tiers)                       |
| PRD trace          | System prompt Section 10 "Security → JWT"      |
| Measurement method | Integration test with Auth Service stopped: existing valid JWTs continue to authenticate for up to 5 min (cache TTL); Jaeger trace shows no span to Auth Service per request |
| Anti-pattern named | "Auth service as synchronous dependency" — if every service calls Auth to validate each token, Auth becomes a T1 dependency for every service and a single point of failure |

### NFR-SEC-010 — JTI Revocation via Redis Blocklist
| Field              | Value                                          |
|--------------------|------------------------------------------------|
| Threshold          | On logout, the JWT's JTI is added to Redis with TTL equal to remaining token lifetime; subsequent requests bearing that JTI return 401 |
| Applies to         | Auth Service (T1); Redis (shared)              |
| PRD trace          | FR-AUTH-005 acceptance criteria                |
| Measurement method | Integration test: login → get token → logout → use token → 401 |

### NFR-SEC-013 — Secrets in Vault Only
| Field              | Value                                          |
|--------------------|------------------------------------------------|
| Threshold          | No secret (DB password, JWT private key, API key, HMAC secret, SMTP credential) may appear in source code, `.env` files, Docker images, or Kubernetes manifests as plaintext; all secrets are read from HashiCorp Vault at runtime |
| Applies to         | All services (All tiers)                       |
| PRD trace          | System prompt Section 10 "Security → Secrets"  |
| Measurement method | gitleaks + trufflehog run on every commit; CI build fails on any detected secret; Vault audit log confirms secrets read at service startup |

## 9. SLO Definitions and Error Budget Policy

### 9.1 SLO Measurement Windows

SLOs are measured over a rolling 28-day window (not calendar month)
to avoid month-boundary effects. The Prometheus recording rules
below are the source of truth — not any application-level metric.

```yaml
# SLO recording rule (example for T1 services)
- record: slo:service_availability:rate28d
  expr: >
    1 - (
      sum(increase(http_requests_total{status=~"5.."}[28d]))
      /
      sum(increase(http_requests_total[28d]))
    )
```

### 9.2 Error Budget Policy

| Budget Remaining | Action Required                                      |
|------------------|------------------------------------------------------|
| > 50 %           | Normal operations; innovation work proceeds          |
| 25 – 50 %        | Review open bugs and reliability work; de-risk before new features |
| 10 – 25 %        | Freeze feature work; reliability sprint mandatory    |
| < 10 %           | Incident review required; no deployments until budget recovers |

### 9.3 SLO Definitions Per Tier

**T1 — Auth, Order, Payment, Inventory**
- Availability SLO: 99.9 %
- Latency SLO: p99 < 500 ms for all read endpoints; p99 < 2 000 ms for write endpoints
- Error budget (28-day): 40 min 19 sec

**T2 — Product Catalog, Cart, Search, Notification, API Gateway**
- Availability SLO: 99.5 %
- Latency SLO: p99 < 500 ms read; p99 < 1 000 ms write
- Error budget (28-day): 3 hr 21 min 36 sec

**T3 — Recommendation, Chatbot, Review, Analytics, Fraud Detection**
- Availability SLO: 99.0 %
- Latency SLO: p99 < 3 000 ms (service-specific overrides per NFR-PERF-*)
- Error budget (28-day): 6 hr 43 min 12 sec

## 11. Glossary

| Term            | Definition                                                              |
|-----------------|-------------------------------------------------------------------------|
| p99             | The 99th percentile of response times; 99 % of requests complete within this value |
| Error budget    | The maximum allowable downtime or error rate within the SLO window      |
| Burn rate       | How quickly the error budget is being consumed relative to the SLO window |
| Fail-open       | When a dependency fails, the system continues processing (accepts risk) |
| Fail-closed     | When a dependency fails, the system rejects the operation (rejects risk) |
| At-least-once   | Kafka delivery guarantee: a message is delivered one or more times; consumers must be idempotent |
| Outbox          | A DB table that stores events transactionally alongside business data before Kafka publish |
| Circuit breaker | A pattern that stops calling a failing dependency to allow recovery time |
| SAST            | Static Application Security Testing — analysis of source code          |
| SCA             | Software Composition Analysis — vulnerability scanning of dependencies  |
| JTI             | JWT ID — unique identifier per token, used for revocation tracking      |
| RED metrics     | Rate, Errors, Duration — standard request-level metric pattern          |
| USE metrics     | Utilisation, Saturation, Errors — standard resource-level metric pattern |
